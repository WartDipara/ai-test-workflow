 ## Stage-driven flow (do NOT execute rigidly)

After `open_game_app`, use `get_ocr_summary` to identify the **current stage** from OCR text, then act accordingly.
Stages are **not guaranteed** and may appear in any order — skip what isn't there.

### privacy — Initial privacy / terms dialog
OCR may show 同意/接受/协议等. Tap the agree button to proceed. If absent, move on.

### download — Resource download / hot-update
Game may show a download confirm dialog or multiple channel options — tap any active button to continue.
Call `analyze_screen` if stuck for a long time with no progress. If absent, move on.

### login — Account login screen
OCR shows username/password fields:
1. `get_ocr_summary` to locate field hints → `fill_credential_field(x, y, username)` → `fill_credential_field(x, y, password)`.
2. Read **VERIFY** lines; re-OCR and re-fill if FAILED.
3. After password filled, login button is pressed automatically by runtime.
See Account/password details below. If no login fields visible, move on.

### server_select — Sub-account / character / server list
OCR shows a list of sub-accounts, characters, or servers. Tap any one to continue (choice doesn't matter). If absent, move on.

### announcement — Post-login popup (semi-transparent overlay)
After login / server select, a semi-transparent announcement popup with mask overlay may appear.
OCR may see both the announcement text and "开始游戏" behind it. Call **`dismiss_overlay`** to auto-dismiss it.
Ignore tiny checkboxes like "今日不再提示" / "以后不再显示" — they are not needed since the game is reinstalled every run.

### privacy_agree — Final privacy terms before "开始游戏"
OCR shows "开始游戏" + privacy / agreement text ("已阅读并同意", "同意隐私条款" etc.) with an unlabeled checkbox.
**Do NOT tap 开始游戏 directly.** This is the **only** scenario worth calling detect_checkbox:
1. Call **`detect_checkbox(prompt="Point to the checkbox")`** → get (x, y).
2. `tap_coordinate(x, y)` to check it (visual state changes to checked).
3. Now tap 开始游戏 to enter. If no checkbox visible, move on.

### in_game — Confirm entry
Call **`check_in_game`** repeatedly until `data.confirmed=true`.

## Account/password login (secure keyboard + self-check)

1. Before keyboard: `get_ocr_summary` for field hint coords → `fill_credential_field(x, y, username)` — **accessibility only** (secure keyboard blocks OCR). Read **VERIFY** lines.
2. If `VERIFY username: FAILED` or `position: FAIL` → **re-OCR** both fields (coords may be wrong) and re-fill; optional `verify_credential_field`.
3. On login screen (keyboard down): `get_ocr_summary` once to **cache Login** coords. `fill_credential_field(x, y, password)` → accessibility only; on VERIFY pass runtime tries **ENTER → u2 Login → cached coords** (black screencap ≠ game freeze). Optional dismiss is one fallback only.
4. If password VERIFY is PARTIAL/FAILED → `verify_credential_field` or `get_ocr_summary` before Login tap.
5. `tap_coordinate` on 登录/Login only after VERIFY passed or PARTIAL with position OK.

Do not tap Login while VERIFY shows wrong field or empty username.

`credentials.yaml` via uiautomator2 only. Device setup: `python -m uiautomator2 init`.

## General strategy tips

- `get_ocr_summary` / `tap_and_observe` use **live device screencap** OCR. Lines in **Keyboard/IME** section (high y) are NOT login buttons — never tap 下一项/符/123 for Login.
- `tap_and_observe` for most UI taps; **`detect_checkbox` only for the privacy checkbox before "开始游戏"** — YOLO is relatively expensive, skip for trivial checkboxes ("今日不再提示" etc.).
- Call **`analyze_screen`** for ambiguous overlays: semi-transparent announcement, mask + background text mixed together, or when OCR result doesn't clearly match a single stage. The multimodal model describes the full screen visually — use that description to decide next action.
- Call **`check_in_game`** after stage 5.
